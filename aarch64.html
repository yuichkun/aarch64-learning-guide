A Field Manual
for AArch64
on Apple Silicon —— M1 のための、機械語とその手触り

本書は M1 / M2 / M3 世代の Apple Silicon で、AArch64 の機械語を 実際に書き、組み立て、走らせ、逆アセンブルして読むための一冊の手引きである。 追加インストールは最小、掲載されるコードはすべてコピーすればターミナルで動く。 読者は自分の指で mov と書き、CPU の中で何が起きているかを確かめる。

序

機械語を「見る」のではなく
組み立てるために

機械語はしばしば「0 と 1 の並び」と説明される。それは正しいが、 書いて、走らせて、逆アセンブルしない限り、手触りは決して得られない。 本書は概念を繰り返し説明する代わりに、M1 の実機の上で動く最小のプログラム を起点に据える。

読む前に、語彙の地図

これから何度も出てくる言葉を、先にひとまとめにしておく。厳密な定義ではなく、 読み進めるための目印として使ってほしい。

AArch64
64ビット実行状態の ARM アーキテクチャ。ARM64 と呼ばれることも多い。iPhone 5s 以降・M1 以降の Mac・多くのサーバで使われている命令セット。
ISA
Instruction Set Architecture。CPU が理解する命令の種類と書式の仕様。AArch64 はひとつの ISA。
命令 (instruction)
AArch64 ではすべての命令が正確に 4 バイト (32 ビット)に収まる固定長。x86 のように可変長ではない。例えば add x0, x0, x1 という 1 行は 32 ビットの数値にエンコードされる。
ニーモニック
命令を覚えやすい短い単語で表したもの (mov, add, ret など)。アセンブラがこれを数値のバイト列に変換する。
レジスタ
CPU 内部の超高速な記憶枠。AArch64 には 64 ビット幅のものが 31 本 (X0〜X30) + 特殊なもの数本。計算はほぼすべてレジスタ同士で行う。
即値 (immediate)
命令の中に直接埋め込まれた定数。# を付けて書く。例: mov x0, #42 の #42。
メモリ
レジスタ外の大きな記憶領域。ldr (load) で読み込み、str (store) で書き込む。AArch64 はレジスタとメモリ間の移動しか命令で行えない (Load/Store アーキテクチャ)。
アセンブラ
.s に書かれた人間向けのニーモニックを、CPU が食うバイト列 (.o) に変換するツール。macOS では as、あるいは clang が裏で呼ぶ。
リンカ
複数の .o を一つの実行ファイルにまとめるツール。_printf のような外部関数の実体 (libc) を繋ぐのもリンカの仕事。
ABI
Application Binary Interface。関数呼び出しでどのレジスタに引数を入れ、どこに戻り値を置くか、スタックをどう揃えるか、といった取り決め。AArch64 には AAPCS64 という標準があり、Apple はそれに独自拡張を加えた Darwin ABI を使う。

ビットとバイト、そして「幅」

上の定義で「64 ビット幅」「32 ビット」という言葉が何度も出てきた。ここで一度、その中身を作っておく。これが分からないと以降の全ての話が宙に浮く。

ビット — 最小の記憶

コンピュータの内部には無数の小さな電子回路がある。そのひとつひとつは、電気が流れているか流れていないかの二つの状態しか取れない。この 2 択の状態を記録できる一番小さな場所を 「1 ビット」と呼ぶ。

電気が流れていない状態を 0、流れている状態を 1 と書くと決めたのが、現代のコンピュータが 0 と 1 だけで全てを表現している理由。電球 1 個に点灯・消灯の 2 状態があるのと同じで、1 ビット単独では「2 通りのうちどちら?」しか表せない。

0電気が流れていない
または
1電気が流れている
1 ビット = 2 通りの状態を区別できる

複数のビットを横に並べると、表現できる幅が広がる

1 ビットだけでは 2 通りしか表せない。でも、2 ビットを横に並べて一組として扱うと、 00 / 01 / 10 / 11 の 4 通りを区別できる。3 ビットなら 8 通り、4 ビットなら 16 通り。 ビット 1 本増えるたびに、表現できる状態数が倍になっていく。

こうして 「何ビットを一組として扱うか」を決めると、その組でいくつの違うものを区別できるかが決まる。 この「一組にするビットの本数」のことを ビット幅 (bit width) と呼ぶ。 「幅」と言うのは、ビットを横一列に並べて眺めたときの列の長さをイメージしているから。

たとえば 8 ビット幅なら、00000000 から 11111111 までの 28 = 256 通り を区別できる。 この 256 という数は、ちょうど以下のような「1 つの塊で扱いたいもの」を 1 個分収めるのに都合が良い:

  • アルファベットや記号 1 文字 (ASCII では全部で 128 種類、収まる)
  • 0〜255 の小さな整数 1 個
  • 画像のピクセル 1 つ分の明るさ (真っ暗 0 〜 最大 255)

だから、ちょうど「何か 1 つを扱うのに使いやすいビット数」として 8 ビットが選ばれた。 この 8 ビットという単位に、歴史的経緯から バイト (byte) という別名が付いた。 以降、1 バイト = 8 ビットと覚えておけばよい。

もっと大きな値を扱いたければ、もっと幅を広げる

1 バイト (8 ビット) では 256 通りしか表せないので、大きな数や複雑なものは扱えない。 そこで用途に応じて、16 ビット・32 ビット・64 ビットという広い幅をまとめて一組として扱う。 下の図は、ビット幅と「区別できる状態の数」の対応を視覚的に並べたもの。 色付きのマスが 1、灰色が 0。

32 ビットもあれば約 42 億種類のものを区別できる。そして 64 ビットあれば約 1844 京通り で、実用上はまずサイズが足りなくなることのない大きさが扱える。AArch64 が「64 ビット」を名乗っているのは、CPU の内部で値を扱う基本単位がこの 64 ビット幅だから、という意味。

この先「ビット幅」という言葉が出てきたら、「何本のビットをひとまとめに扱うか」と読み替えればよい。 「X レジスタは 64 ビット幅」というのは、後の章で登場する X レジスタという部品が、64 本のビットを一組として扱える場所だ、という意味にすぎない。

本書の流儀

扱うのは AArch64 (別名 ARM64、ARMv8-A の 64bit 実行状態)。 iPhone 5s 以降の Apple 製 SoC と、M1 以降の Mac はすべてこの ISA で動いている。

本書は読者が 3 通りの「機械語の呼び方」 を区別できるようになることを目標とする。 すなわち、libc を呼ぶアセンブリ (本筋)、 C の出力としてのアセンブリ (比較対象)、 syscall を直接叩くアセンブリ (補助) の三つ。

コードは すべて「何をしているか」を 1 行ずつ説明 する。 「動いた」で終わらせず、「なぜ動くか」に必ず立ち戻る。

Hello, Silicon — 動作確認

下のコマンドを一括で流す。Hello, Silicon と表示されれば環境は整っている。 このコード自体の逐行解説は、必要な語彙 (レジスタ・命令) を揃えた後の第五章で行う。いまは動くことだけ確認すれば十分。 

# asm を書いて、組み立てて、リンクして、走らせる
cat > hello.s <<'EOF'
.global _main
.align 2
_main:
    stp   x29, x30, [sp, #-16]!
    adrp  x0, msg@PAGE
    add   x0, x0, msg@PAGEOFF
    bl    _puts
    mov   w0, #0
    ldp   x29, x30, [sp], #16

.data

第二章 · 三つのアプローチ

同じ「Hello」でも
書き方は三通りある

同じ出力を得るのに、機械語への降り方は複数ある。本書では以下の三つを使い分ける。 基本は 方針 B（libc を呼ぶ） で進め、 コンパイラの出力を見るための 方針 C を並走、 OS の境界を覗くときのみ 方針 A（syscall） に降りる。

B

アセンブリから libc を呼ぶ

自分で書いたアセンブリが puts、printf、 malloc を呼ぶ方式。Apple がこれを推奨しており、 書ける範囲が広く、教材も豊富。本書の主軸。

C

C を書いて asm 出力を見る

clang -S や clang -O2 -S で、 C コードがどんな AArch64 に落ちるかを観察する方式。 自分で書いた asm との比較で、最適化や呼び出し規約が一気に理解できる。

A

syscall を直接叩く

svc #0x80 で Darwin のシステムコールを直接呼ぶ最低レベル方式。 libc を介さない分、OS の境界が露出する。本書では補助として紹介する。

第三章 · レジスタ早見表

31 本の汎用レジスタと
特別な数本

そもそもレジスタとは何か

これまで X0 や W0 といった名前が何度も出てきたが、ここで一度ちゃんと定義する。 レジスタとは、CPU のチップ内部に埋め込まれた、非常に小さくて非常に速い記憶枠のこと。 メモリ (RAM) がチップの外にある広大な倉庫だとすれば、レジスタは CPU の作業机の上に最初から置かれている引き出しに近い。

CPU は原則として、メモリ上の値を直接計算することはできない。 足し算一つするにも、まず メモリからレジスタへ値を読み出す → レジスタ同士で計算 → 結果をメモリへ書き戻す、という手順を踏む。 この「計算の一時待機所」がレジスタで、数が多いほど、そして速いほど CPU の仕事が速くなる。

ただし、レジスタは非常に高価な部品なので、本数はごく少ない。 AArch64 では 31 本 + 特殊なもの数本。これを ABI のルールに従って使い回していく。

レジスタ 1 本分のサイズ = 64 ビット幅

前章で扱った「ビット幅」の話がここで効いてくる。 AArch64 のレジスタ 1 本は 64 ビット幅 ── つまり、0 と 1 を 64 個分記録できる箱が 1 本。 この箱の中に、64 ビットで表現された任意の値 (約 1844 京通りのうちのどれか) を 1 つ格納できる。

本章ではこの 31 本のレジスタの名前と役割を眺めていく。 まず、1 本のレジスタに2 つの呼び名があるという話から始める。

同じレジスタ、2 つの名前 — X と W

AArch64 のレジスタは 64 ビット幅だが、時には 下位 32 ビットだけを扱いたい場面がある。 たとえば C の int 型は 32 ビット幅なので、64 ビット全部を使う必要がない。 そこで、同じ 1 本のレジスタに対して 2 通りの呼び方が用意されている:

  • X0, X1, … X30 — 64 ビット全体を指す呼び方 (X は 64bit extended の X)
  • W0, W1, … W30 — 下位 32 ビットだけを指す呼び方 (W は 32bit word の W)

X0 と W0 は同じ物理レジスタの、扱う幅だけが違う呼び方。 別々のレジスタではない。

X レジスタと W レジスタの関係
物理的には 1 本

X0 と W0 は同じ箱の呼び方違い。W レジスタに書き込むと、上位 32 ビットは自動的にゼロクリアされる。

書くときの使い分け

扱いたい値が 32 ビット以内に収まるなら W、それ以上なら X。上位 32 ビットを使わないなら W で十分。

例
mov w0, #42    → X0 全体が 0x00000000_0000002A
add w0, w0, w1 → 32bit 加算、上位は 0 に

C の型との対応 — なぜ main の return 0; で w0 なのか

「ビット幅」と「X/W レジスタ」が揃ったところで、これまで保留してきた疑問に答えられる。 C で書かれた関数の戻り値は、その型のビット幅に応じて X か W のどちらかで返すことになっている。 型のビット幅と使うレジスタの対応はこうなる:

char
8 ビット幅 = 1 バイト。W レジスタの下位 8 ビット部分だけ使う。
short
16 ビット幅 = 2 バイト。W レジスタの下位 16 ビット部分を使う。
int
32 ビット幅 = 4 バイト。W レジスタをちょうど満たすサイズ。だから int を返す関数は w0 を使う。
long / ポインタ
64 ビット幅 = 8 バイト (macOS・Linux の場合)。X レジスタをちょうど満たすサイズ。だから long や ポインタを返すときは x0。

これが、第一章の Hello, Silicon で mov w0, #0 と書いた理由。 C の main() の戻り値型は int (32 ビット) なので、レジスタの 64 ビット全部ではなく下位 32 ビット分にあたる w0 に書き込む ── これが流儀。

誰が何を保存する責任を持つか

関数呼び出しをまたいでレジスタの値を守るのは、呼び出す側か、呼ばれる側か。 これを 「保存責任 (save convention)」と呼び、ABI で決まっている。

Caller-saved
呼び出す側の責任。X0〜X18 の多く。関数を呼ぶ前に、残しておきたい値は自分でスタックに退避する。呼ばれた側は自由に書き換えてよい。引数と戻り値、一時変数に向く。
Callee-saved
呼ばれた側の責任。X19〜X28 の 10 本。関数内で使うならプロローグで退避、エピローグで復元する義務がある。呼び出しをまたいで保持したい値に向く。自分がループ内で何度も関数を呼ぶなら、こちらに値を入れる。
特殊
X29 (FP) と X30 (LR) はフレームポインタとリンクレジスタ。X18 は Apple が予約済み (OS が使う) で触ってはいけない。SP はスタックポインタで、常に 16 バイト境界に揃える必要がある。

以下のグリッドは X0 から X30 と特殊レジスタの配置を示している。セルをクリックすると右に詳しい説明が出る。 実務では X0〜X7 (引数)、X29/X30 (フレーム)、X19〜X28 (呼び出しまたぎの保管庫) を覚えれば大体足りる。

X0

General Purpose · Argument 0 / Return Value

汎用レジスタの 0 番。呼び出し規約 (AAPCS64) では第 1 引数と戻り値に使われる。 C 関数の printf(fmt, ...) を呼ぶときは X0 に第 1 引数（フォーマット文字列のアドレス）を置く。 Darwin では syscall の第 1 引数兼戻り値としても使われる。

フラグ (NZCV)

通常のレジスタとは別に、4 ビットの「条件フラグ」がある。cmp や adds などの命令で更新され、条件分岐命令 b.eq / b.gt などがこれを見て飛ぶかどうかを決める。

N (Negative)
結果が負 (最上位ビットが 1) なら 1。
Z (Zero)
結果がちょうど 0 なら 1。「等しいか?」の判定に使う。b.eq は Z=1 のとき飛ぶ。
C (Carry)
符号なしでのキャリー / ボロー (繰り上がり / 繰り下がり)。cmp a, b で a ≥ b (符号なし) なら 1。
V (oVerflow)
符号ありで溢れが起きたら 1。b.lt / b.gt の判定に効く。
第四章 · 命令リファレンス

最小限の
語彙

AArch64 は RISC 寄りの単純な ISA で、実用的な範囲なら 50 命令ほど知っていれば読み書きできる。 以下はその中核。ニーモニック・書き方・意味・補足の 4 つを併記してある。 命令の書式は mnemonic Xd, Xn, Xm のように「左から:置き先, ソース 1, ソース 2」の順番が基本、と覚えるとほぼ読める。

命令の読み方 — ほぼ全てこの形

X ではなく W で書くと 32bit 版になる。例: add w0, w1, w2 → 32bit 加算。

教材で見る Xd, Xn, Xm という記法は「d = destination (置き先)、n, m = source (ソース)」の慣習的な名付け。

第五章 · Hello, Silicon を解剖する

11 行の中に詰まっている
すべて

第一章で動かした Hello, Silicon のコードを、語彙が揃った今、改めて 1 行ずつ読み解く。 この 11 行に AArch64 と Apple ABI の基本がほぼ全て詰まっている。 ここを押さえれば、以降のハンズオンで出てくるパターンの大部分が既知のものになる。 

// hello.s ── C の printf 相当を asm だけで書いたもの
.global _main
.align 2
_main:
    stp   x29, x30, [sp, #-16]!
    adrp  x0, msg@PAGE
    add   x0, x0, msg@PAGEOFF
    bl    _puts
    mov   w0, #0
    ldp   x29, x30, [sp], #16
    ret
.data
msg: .asciz "Hello, Silicon"
hello.s · 逐行解説 AArch64 · Darwin ABI
.global _main
「_main というシンボルは外から見える」 とリンカに伝える指示 (アセンブラ・ディレクティブ)。.global を付けないとリンカから見つからずリンクエラーになる。Apple の慣習で C の main はアセンブリ側では _main と頭にアンダースコアが付く。
.align 2
次に置くコードを 22 = 4 バイト境界 に整列させる指示。AArch64 の命令は全て 4 バイト固定長なので、命令の先頭アドレスが 4 の倍数になっていないと CPU がまともに実行できない。.align を忘れるとクラッシュする。「整列」「境界」って何？は この直後の図解を参照。
_main:
ラベル。ここから下の命令列の先頭アドレスに _main という名前を付ける。Cランタイムのスタートアップコードはこのラベルをエントリーポイントとして呼び出す。
stp x29, x30, [sp, #-16]!
関数の冒頭で必ず書く プロローグ。stp は Store Pair、2つのレジスタを同時にメモリに保存する命令。ここでは x29 (フレームポインタ) と x30 (リンクレジスタ=戻り先アドレス) をスタックに退避している。[sp, #-16]! は「sp から 16 を引いて、その位置にストアして、その結果を sp に書き戻す」(プレデクリメント・ライトバック)。なぜ必要か: この後 bl _puts で関数呼び出しをすると x30 が上書きされてしまうので、先に避難させておく。
adrp x0, msg@PAGE
adrp = "Address of PaGe"。msg という文字列が置かれている 4KB ページの先頭アドレスを x0 にロードする。なぜ 2 段階で住所を作るのかというと、AArch64 の 1 命令は 32 ビットしか情報が入らず、64 ビットの絶対アドレスをそのまま埋め込めないから。そこで「ページ単位で大まかに飛んで、次の命令で端数を足す」という設計になっている。
add x0, x0, msg@PAGEOFF
さっきのページ先頭に、msg のページ内オフセット (0〜4095) を足して、正確な文字列アドレスを完成させる。結果として x0 には "Hello, Silicon" の先頭アドレスが入る。X0 は AAPCS64 で第 1 引数として使われるレジスタなので、これは次の puts への第 1 引数を準備している、ということ。
bl _puts
bl = Branch with Link。関数呼び出しの命令。動作は 2 つあって、(1) この命令の次のアドレスを x30 に書き込む (= 戻り場所を覚える)、(2) _puts のアドレスへジャンプする。_puts の実体はリンカが libc から解決する。C で書けば puts("Hello, Silicon"); 一行。
mov w0, #0
C の return 0; に相当。w0 は 32 ビット幅の戻り値用レジスタで、C の int 型の戻り値 (0) をここに入れる。これで「正常終了」が呼び出し元 (Cランタイム) に伝わる。 w0 と x0 の違いについては第三章「レジスタ」で詳述。ここでは「int を返すから w0 を使っている」とだけ覚えておけば十分。
ldp x29, x30, [sp], #16
関数末尾の エピローグ。プロローグで退避した 2 つのレジスタをスタックから復元する。ldp = Load Pair。[sp], #16 は「現在の sp からロードして、その後 sp に 16 を足す」(ポストインクリメント)。プロローグの逆操作でスタックを元に戻している。
ret
関数からの復帰。暗黙に x30 (リンクレジスタ) の値をプログラムカウンタにロードしてジャンプする。つまり bl _main で呼んできた Cランタイムの「続き」に戻る。エピローグで x30 を正しく復元しておかないと、どこか変な場所に飛んでクラッシュする。
.data
ここから下はコードではなくデータセクションですよ、という指示。リンカがこの部分を実行ファイルの「データ領域」(読み書き可能、実行不可) に配置する。
msg: .asciz "Hello, Silicon"
ラベル msg を付けた場所に、ナル終端の ASCII 文字列を置く。.asciz の z は "zero" の z、つまり自動的に末尾に \0 バイトを追加する。C の文字列と同じ仕様で、puts はこの \0 を見て終端を判断する。
ここまでで分かること
AArch64 プログラムは、(1) プロローグで x29/x30 を退避 → (2) 引数をレジスタに用意 (第1引数は x0) → (3) bl で関数呼び出し → (4) 戻り値を x0/w0 に入れる → (5) エピローグで x29/x30 を復元 → (6) ret という流れが基本。これは今後出てくる全ての関数で同じパターンなので、最初にしっかり覚える価値がある。

.align N ディレクティブは、次に置くデータの先頭アドレスを 2N バイト境界に揃えるという意味。必要なら間に 0 (NOP) を自動で詰めてくれる。

.align 3

.align 4

つまり .align 2 は 「この先にあるものは 4 バイト境界に置いてね」とアセンブラに頼む、たった一行のおまじない。 これを忘れると、直前にたまたま文字列データなどが 4 の倍数で終わっていなかった場合にズレて、CPU が動かない。 AArch64 アセンブリを書くときは、関数の直前に必ず .align 2 を書く、と覚えてしまえばよい。

前章で Hello, Silicon の構造を押さえた今、手を動かして実際に小さなプログラムを書き足していく番。 本章ではコピー & ペーストで動くサンプルが並ぶ。方針 B（libc 呼び出し）、 方針 C（C から asm を見る）、方針 A（syscall 直叩き）の三系統。 いずれも作業ディレクトリで cat > file.s <<'EOF' の要領で投入し、 clang -o でビルドする。

B-1 · 足し算関数を asm で書き、C から呼ぶ

最も典型的な「asm + C」の形。add_two という関数をアセンブリで書いて、C から呼び出す。この 5 行だけで AArch64 のすべてが詰まっているので、じっくり見る。

// add.s ── x0 と x1 を足して x0 で返す
.global _add_two
.align 2
_add_two:
    add   x0, x0, x1
    ret
add.s · 逐行解説5 lines
.global _add_two
関数 _add_two を外部公開する (C から呼べるようにする)。C の add_two はアセンブリ側では _add_two、先頭にアンダースコアが付く Apple の慣習。
.align 2
次に続く命令を 4 バイト境界に揃える。AArch64 命令は 4 バイト固定長なので必須。
_add_two:
関数本体の入口。C から呼ばれるとここに飛んでくる。このとき ABI により、a は x0 に、b は x1 に、それぞれ既に入っている状態になっている。
add x0, x0, x1
意味は x0 = x0 + x1。左が結果の置き先、右 2 つが足すソース。結果を x0 に入れるのは x0 が戻り値用のレジスタだから (AAPCS64)。C で言えば return a + b; の本体。この関数は x30 を壊さない (別の関数を呼ばない) ので、プロローグ／エピローグが不要。これが「葉関数 (leaf function)」と呼ばれる最小形。
ret
呼び出し元に戻る。暗黙に x30 の値へジャンプする。C から呼ばれたときは、C コンパイラが用意した戻り先が x30 に入っているので、そこに戻れば関数としての務めは完了。
// main.c
#include <stdio.h>
long add_two(long a, long b);       // asm 側を宣言
int main(void) {
    printf("%ld\n", add_two(40, 2));// C から呼ぶ
    return 0;
}
clang -o add main.c add.s
./add    # → 42
肝
C 側が add_two(40, 2) と書くと、C コンパイラが「40 を x0、2 を x1 に入れて bl _add_two」というコードを吐く。アセンブリ側はそのお膳立て済みの状態でスタートする。だから asm 側で x0 と x1 の初期値の出所を気にしなくていい。これが ABI の力。

B-2 · ループで合計を計算する

1 から n までを足す。ループと条件分岐をどう書くかがここで分かる。

// sum.s ── 1 から x0 までの整数を合計して x0 で返す
.global _sum_to
.align 2
_sum_to:
    mov   x1, #0            // accumulator
    mov   x2, #1            // counter
Lloop:
    cmp   x2, x0
    b.gt  Ldone
    add   x1, x1, x2
    add   x2, x2, #1
    b     Lloop
Ldone:
    mov   x0, x1
    ret
sum.s · 逐行解説ループ構造の分解
mov x1, #0
C の long s = 0; に相当。累積和を入れる箱を x1 に用意し 0 で初期化。x1 は呼び出し元から見れば引数だったレジスタだが、sum_to は引数を 1 個しか取らない (x0 のみ使用) のでここでは自由に使える。
mov x2, #1
ループカウンタ i を x2 に用意し 1 からスタート。C で言えば for (long i = 1; ...) の i = 1 の部分。
Lloop:
ループ先頭のラベル。ここへ戻ってくる。L で始まるラベルはローカル (関数内限定) の慣習。
cmp x2, x0
比較命令。中身は「x2 - x0 を計算するが結果は捨て、フラグ (NZCV) だけ更新する」。つまり i - n の符号と零判定を準備している。結果自体はどこにも残らない。
b.gt Ldone
Branch if Greater Than。直前の比較結果が「>」なら Ldone へ飛ぶ。C で言えば if (i > n) goto Ldone;。つまり「カウンタが n を超えたら終了」。フラグが gt 条件を満たさなければ、次の命令へ素通りする。
add x1, x1, x2
s += i。累積和に現在のカウンタ値を加える。
add x2, x2, #1
i++。カウンタを 1 増やす。レジスタと即値の加算なので、2 つ目のソースに #1 が来ている。
b Lloop
無条件で Lloop へジャンプ。C の「ループの末尾から先頭へ戻る」に対応。b は bl と違い x30 を書き換えない (戻ってくる予定がないので)。
Ldone:
ループ脱出地点。
mov x0, x1
累積和を戻り値レジスタ x0 にコピー。なぜ最初から x0 に溜めなかったかというと、x0 は引数 (n) で使われていたから。ループ条件で x0 と比較するので、書き換えずに取っておく必要があった。
ret
戻る。この関数も他の関数を呼ばないのでプロローグ／エピローグ不要。
// main.c
#include <stdio.h>
long sum_to(long n);
int main(void) { printf("%ld\n", sum_to(100)); return 0; }

# ビルドと実行
clang -o sum main.c sum.s && ./sum   # → 5050
学び
AArch64 の条件分岐は必ず 2 段階: (1) cmp で比較してフラグを立てる → (2) b.xx でそのフラグを見て飛ぶ。cmp の結果をどこにも保存しないのが奇妙に感じるかもしれないが、フラグという「隠れた出力」があると考えるとよい。

B-3 · スタックフレームと printf の呼び出し

関数から printf を呼ぶには、x30 (LR) を退避するためのスタックフレームを作る必要がある (B-1, B-2 は他の関数を呼ばなかったので不要だった)。 Apple ABI では stp x29, x30, [sp, #-16]! が定型句。 このサンプルは冒頭の Hello, Silicon と同じ構造だが、引数を 3 つ渡す部分が新しい。 

// greet.s ── printf("%s, %ld!\n", name, n)
.global _main
.align 2
_main:
    stp   x29, x30, [sp, #-16]!   // プロローグ
    mov   x29, sp

    adrp  x0, fmt@PAGE
    add   x0, x0, fmt@PAGEOFF
    adrp  x1, name@PAGE
    add   x1, x1, name@PAGEOFF
    mov   x2, #42
    bl    _printf

    mov   w0, #0
    ldp   x29, x30, [sp], #16    // エピローグ
    ret

.data
fmt:  .asciz "%s, %ld!\n"
name: .asciz "Hello"
greet.s · 新しい部分だけ解説printf への引数 3 つの渡し方
mov x29, sp
新しいフレームポインタを設定する。プロローグで sp を 16 下げたので、その新しい sp を x29 (FP) に記録しておく。デバッガが関数のフレームを辿るときに使う。省略しても動くが、書くのが作法。
adrp x0, fmt@PAGE
add x0, x0, fmt@PAGEOFF
printf の第 1 引数 (フォーマット文字列) を x0 に。adrp+add の 2 段で絶対アドレスを組み立てる (Hello, Silicon と同じ)。
adrp x1, name@PAGE
add x1, x1, name@PAGEOFF
printf の第 2 引数 ("%s" に対応する文字列) を x1 に。第 2 引数は x1 というのが ABI のルール。
mov x2, #42
printf の第 3 引数 ("%ld" に対応する数値) を x2 に。第 3 引数は x2。即値 (定数 42) なので mov で直接入れる。
引数は x0, x1, x2, x3, ... x7 の順に 8 個までレジスタ渡し、9 個目以降はスタック経由になる。
bl _printf
printf を呼ぶ。この瞬間 x30 は書き換えられる (bl の (1) の動作) のだが、プロローグで退避済みなので問題ない。printf の中で x0〜x2 も壊される可能性があるが、この関数ではもう使わないので OK。

C-1 · C を書いて asm を見る

clang -S で .s を生成すると、コンパイラがどんな AArch64 を吐くかを観察できる。 最適化レベルを変えて比較すると効果が一目瞭然。 

// square.c
long square(long x) { return x * x; }
long sum_to(long n) {
    long s = 0;
    for (long i = 1; i <= n; i++) s += i;
    return s;
}
# 最適化なし（素朴な対応が見える）
clang -S -O0 -o square-O0.s square.c

# 最適化あり（ループが閉じた式になるかもしれない）
clang -S -O2 -o square-O2.s square.c

# 差分を見る
diff square-O0.s square-O2.s | head -60

A-1 · write syscall で直接出力する

libc を一切使わず、Darwin の write(2) システムコールだけで文字列を出す。 OS との境界がここに露出している、つまり普段は libc が隠している部分を自分で書く。 syscall 番号は X16、引数は X0〜X5、呼び出しは svc #0x80。 

// raw.s ── libc なしで "Raw!\n" を書く
.global _main
.align 2

_main:
    mov   x0, #1             // fd = stdout
    adrp  x1, msg@PAGE
    add   x1, x1, msg@PAGEOFF
    mov   x2, #5             // length
    mov   x16, #4            // SYS_write = 4
    svc   #0x80           // カーネルに降りる

    mov   x0, #0
    mov   x16, #1            // SYS_exit = 1
    svc   #0x80

.data
msg: .ascii "Raw!\n"
raw.s · syscall の仕組みDarwin ABI
mov x0, #1
write(fd, buf, len) の第 1 引数 fd。1 は stdout (標準出力) を意味する慣習的な番号。2 なら stderr、0 なら stdin。
adrp x1, msg@PAGE
add x1, x1, msg@PAGEOFF
第 2 引数 buf = 書き出す文字列の先頭アドレスを x1 に。libc 呼び出しと同じ adrp+add の 2 段構え。
mov x2, #5
第 3 引数 len = 書き出すバイト数。"Raw!\n" で 5 バイト (R, a, w, !, 改行)。
mov x16, #4
ここが syscall と関数呼び出しの違い。Darwin は呼び出したいシステムコールの番号を x16 に入れる慣習。4 は write の番号 (<sys/syscall.h> の SYS_write に定義されている)。Linux では x8 を使うなど OS によって違う。
svc #0x80
SVC = Supervisor Call。これが 「ユーザランドからカーネルへ降りる」瞬間。CPU の特権レベルが切り替わり、カーネルが x16 の番号を見て対応する処理 (write) を実行する。引数 x0〜x5 はカーネル側がそのまま読み、結果を x0 に書いて戻ってくる。#0x80 は Darwin の慣習的な値で、意味は OS が定める。
mov x0, #0
mov x16, #1
svc #0x80
プログラムを終了する syscall。exit(0) に相当。番号 1 が SYS_exit、第 1 引数 (x0) が終了ステータス。libc が普段やっている「main の戻り値を受け取って exit システムコールを呼ぶ」ところを自分で書いている。この svc は戻ってこないのでプロセスが終わる。
clang -o raw raw.s && ./raw    # → Raw!
一段下の世界
B-3 の printf("Hello, 42!\n") は、内部的にはこの A-1 とほぼ同じことをしている ── 文字列をバッファに整形し、最終的には write(1, buf, len) の syscall が呼ばれている。libc は syscall の薄いラッパー、と言われる所以。
なぜ補助扱いか
Darwin の syscall 番号は Apple から公式に一覧が公開されておらず、 /usr/include/sys/syscall.h や XNU カーネルのソース (syscalls.master) を 読まないと正確な番号が分からない。Linux の asm/unistd.h のように 安定した入口ではないので、学習の主軸にするには不向きである。

ここまで読んできた内容で、実はそれなりの範囲のプログラムが自分で書けるようになっている。 6 題、簡単なものから少し手応えのあるものまで並べる。 まずは紙かエディタで自分で書いてみてほしい。 書けたら第七章のシミュレータに貼り付けて Run、あるいは実機で clang -o して実行し、答え合わせをする。 それでも詰まったら「ヒント」を、それでもダメなら「正答を見る」をクリック。

ほとんどの問題は、関数 1 つを書く形式になっている。入力は x0、(あれば) x1 から受け取り、答えを x0 に入れて ret で返すのが標準の決まり。これは AAPCS64 の規則そのものなので、以後 asm で関数を書くたびに使うことになる。

No. 01 ★ やさしい

倍にする

関数 doubled(x) を AArch64 アセンブリで書け。引数 x (x0) を 2 倍した値を返す。シミュレータで mov x0, #21 から始めて、42 が戻り値になれば正解。

条件: 使える命令は add, lsl, mov, ret のどれでも OK。できれば 2 通り以上の書き方を試してみてほしい。

ヒント
2x は「x + x」と考えてもよいし、「x を 1 ビット左シフト」とも考えられる (2 進数で見ると、左シフトは 2 倍)。lsl (logical shift left) は lsl Xd, Xn, #sh の形。
正答を見る

解法 A · 自分自身を足す

    mov   x0, #21        // 入力のセットアップ (動作確認用)
doubled:
    add   x0, x0, x0   // x0 = x0 + x0
    ret

解法 B · 左シフト

    mov   x0, #21
doubled:
    lsl   x0, x0, #1    // x0 = x0 << 1 = x0 * 2
    ret
どちらも 1 命令。コンパイラは状況に応じて使い分けるが、意味は同じ。左シフト 1 回分 = 2 倍、n 回 = 2n 倍という性質は以降よく出てくるので頭の隅に。
No. 02 ★ やさしい

偶数か奇数か

関数 is_odd(x) を書け。x が奇数なら 1、偶数なら 0 を返す。例: is_odd(7) = 1、is_odd(10) = 0。

分岐を使わずに 2 命令以内で書くのが理想。

ヒント
2 進数で「偶数か奇数か」は最下位ビットを見るだけで分かる。最下位ビットが 0 なら偶数、1 なら奇数 (10 進でも「1 の位が偶数か奇数か」で判断するのと同じ要領)。
ビットごとの AND を取ってその 1 ビットだけ残すには、and Xd, Xn, #1 を使えばよい (マスク操作)。
正答を見る

正答

    mov   x0, #7        // 動作確認用: 7 は奇数なので 1 を期待
is_odd:
    and   x0, x0, #1   // 最下位ビットだけ残して他を 0 に
    ret
and で x0 の全 64 ビットのうち最下位 1 ビットだけ残し、他を 0 にしている。結果として最下位が 1 (奇数) なら 1、0 (偶数) なら 0 が x0 に入る。分岐一切なしで判定できるのがこの手の「ビット演算による条件計算」の美しさ。
No. 03 ★★ ふつう

フィボナッチ

関数 fib(n) を書け。入力 n (x0) に対して、n 番目のフィボナッチ数を返す。

数列の定義: fib(0) = 0, fib(1) = 1, fib(n) = fib(n-1) + fib(n-2)。例: fib(10) = 55。

条件: 再帰は使わない。ループで書く (再帰は関数呼び出しが入って難易度が跳ね上がる)。

ヒント
3 本のレジスタで十分。前の値 a と現在の値 b を保持して、毎ループで (a, b) → (b, a+b) と更新していく。
注意: 値の入れ替えをするときは一時レジスタが必要。a = b; b = a + b; を素直に書くと b が壊れるので、tmp = a + b; a = b; b = tmp; のような順で書く。
ループの継続判定には cbz (zero なら分岐) と sub (カウンタを 1 減らす) が便利。
正答を見る

正答

    mov   x0, #10       // 動作確認用: fib(10) は 55
fib:
    mov   x1, #0        // a = 0  (= fib(0))
    mov   x2, #1        // b = 1  (= fib(1))
Lloop:
    cbz   x0, Ldone    // n == 0 なら終了
    add   x3, x1, x2   // tmp = a + b
    mov   x1, x2        // a = b
    mov   x2, x3        // b = tmp
    sub   x0, x0, #1   // n -= 1
    b     Lloop
Ldone:
    mov   x0, x1        // 返り値は a
    ret
ループ回数 = 引数 n の値。毎回 (a, b) を 1 つ先に進め、n 回進んだあとの a が答え。
ポイント: 入力 n は x0 に入るが、ループカウンタとしてそのまま使って問題ない (消費しながら減らしていく)。最後に答えを x0 に入れ直して ret。ループカウンタと戻り値レジスタが同じというのは asm の面白さ。
No. 04 ★★ ふつう

絶対値

関数 my_abs(x) を書け。x の絶対値を返す。例: my_abs(-7) = 7, my_abs(5) = 5。

葉関数として書くこと。つまり stp / ldp のプロローグ・エピローグを書かない。

ヒント
ゼロとの比較 (cmp x0, #0) でフラグを立て、負なら neg で符号反転。正ならそのまま。
b.ge (greater or equal = Z=1 または N=V) を使うと「0 以上なら飛ぶ」条件分岐になる。
正答を見る

正答

    mov   x0, #-7       // 動作確認用: -7 の絶対値は 7
my_abs:
    cmp   x0, #0
    b.ge  Lpos         // x0 >= 0 ならそのまま返す
    neg   x0, x0        // 負なら符号反転
Lpos:
    ret
葉関数 (他の関数を呼ばない関数) なので x30 を壊す心配がなく、プロローグ/エピローグは不要。プロローグを省けるのは「自分が bl を使わない」ときだけ。
No. 05 ★★★ しっかり考える

配列の合計

関数 sum_array(ptr, n) を書け。引数 ptr (x0) は 64 ビット整数の配列の先頭アドレス、n (x1) は要素数。全要素の合計を返す。

ポイント: メモリからのロードと、アドレスを進めながらのループが要る。

シミュレータで試すには、先に str で適当な値を何個か置いてから sum_array を呼び出す形にするとよい (シミュレータ例題「メモリ書き込み」を参考に)。

ヒント
  • ldr x3, [x0] で「x0 が指すアドレスから 8 バイト読み込んで x3 に」
  • アドレスを 8 バイト進めるには add x0, x0, #8 (64 ビット整数 1 個分 = 8 バイト)
  • 累積和を x2 など別のレジスタに貯めると分かりやすい
  • ループ終了判定は cbz x1, ... (残り要素数が 0 になったら抜ける)
正答を見る

正答

// シミュレータで試すための準備 (3要素の配列を 0x1000 に置く)
    mov   x10, #0x1000
    mov   x11, #10
    str   x11, [x10]
    mov   x11, #20
    str   x11, [x10, #8]
    mov   x11, #30
    str   x11, [x10, #16]

    mov   x0, x10       // ptr = 0x1000
    mov   x1, #3         // n = 3

sum_array:
    mov   x2, #0         // accumulator = 0
Lloop:
    cbz   x1, Ldone     // n == 0 なら終了
    ldr   x3, [x0]      // x3 = *ptr
    add   x2, x2, x3    // acc += *ptr
    add   x0, x0, #8    // ptr += 1  (1 要素 = 8 バイト)
    sub   x1, x1, #1    // n -= 1
    b     Lloop
Ldone:
    mov   x0, x2         // 答えを戻り値レジスタへ
    ret                   // 結果: 10 + 20 + 30 = 60
アドレスもレジスタで扱う値のひとつというのが腑に落ちる問題。「配列を走査する」とは、要するに「アドレスを足し算で進める」こと。
ポインタ型と整数型が機械語レベルでは同じ 64 ビット値であるという事実が、ここで体験できる。
No. 06 ★★★★ 総合

FizzBuzz

1 から 15 までの数について:

  • 3 の倍数なら "Fizz"
  • 5 の倍数なら "Buzz"
  • 両方の倍数 (15 の倍数) なら "FizzBuzz"
  • どれでもない数字ならその数字自体

を出力するプログラムを書け。これまでに出てきた要素のほぼ全てを使う: プロローグ/エピローグ、ループ、条件分岐、printf の呼び出し、adrp/add の 2 段アドレス構築。

ヒント多めなので、まずは「1 から 15 まで数字をそのまま出力する」部分だけ書いて動かし、それから Fizz/Buzz の判定を足していくとよい。

ヒント
  • 「3 の倍数」の判定には除算の剰余がほしい。AArch64 には直接の剰余命令がないが、udiv で商を計算し、a - (a/b)*b で剰余を作れる。msub という命令がこれを 1 命令でやってくれる: msub Xd, Xn, Xm, Xa は Xd = Xa - (Xn * Xm)。
  • 剰余が 0 かどうかは cbz で判定できる。
  • x30 (戻り先アドレス) は printf を呼ぶたびに壊されるので、プロローグで退避する。
  • ループカウンタ (1 から 15 まで) は x19 など callee-saved レジスタに入れておくと、printf 呼び出しをまたいでも保持される。x19 を使うなら、プロローグで退避する必要がある (C 関数のお作法)。
  • 数字を出力するときは printf("%ld\n", n)、文字列を出力するときは printf("%s\n", "Fizz") または単に puts("Fizz")。
正答を見る

正答 · fizzbuzz.s

.global _main
.align 2

_main:
    stp   x29, x30, [sp, #-32]!   // フレーム確保 (16+16 = 32 バイト)
    stp   x19, x20, [sp, #16]     // x19, x20 も退避 (callee-saved)
    mov   x29, sp

    mov   x19, #1                // カウンタ i = 1
Lloop:
    cmp   x19, #15
    b.gt  Ldone                   // i > 15 なら終了

    // i % 15 を計算 (FizzBuzz 判定)
    mov   x20, #15
    udiv  x1, x19, x20         // x1 = i / 15
    msub  x1, x1, x20, x19     // x1 = i - (i/15)*15 = i % 15
    cbz   x1, Lfizzbuzz

    // i % 3 == 0 か
    mov   x20, #3
    udiv  x1, x19, x20
    msub  x1, x1, x20, x19
    cbz   x1, Lfizz

    // i % 5 == 0 か
    mov   x20, #5
    udiv  x1, x19, x20
    msub  x1, x1, x20, x19
    cbz   x1, Lbuzz

    // どれでもない: 数字そのものを出力
    adrp  x0, fmt_num@PAGE
    add   x0, x0, fmt_num@PAGEOFF
    mov   x1, x19
    bl    _printf
    b     Lnext

Lfizz:
    adrp  x0, s_fizz@PAGE
    add   x0, x0, s_fizz@PAGEOFF
    bl    _puts
    b     Lnext

Lbuzz:
    adrp  x0, s_buzz@PAGE
    add   x0, x0, s_buzz@PAGEOFF
    bl    _puts
    b     Lnext

Lfizzbuzz:
    adrp  x0, s_fb@PAGE
    add   x0, x0, s_fb@PAGEOFF
    bl    _puts

Lnext:
    add   x19, x19, #1
    b     Lloop

Ldone:
    mov   w0, #0
    ldp   x19, x20, [sp, #16]
    ldp   x29, x30, [sp], #32
    ret

.data
fmt_num: .asciz "%ld\n"
s_fizz:  .asciz "Fizz"
s_buzz:  .asciz "Buzz"
s_fb:    .asciz "FizzBuzz"

ビルドと実行

clang -o fizzbuzz fizzbuzz.s && ./fizzbuzz

# 2
# Fizz
# 4
# Buzz
# Fizz
# 7
# 8
# Fizz
# Buzz
# 11
# Fizz
# 13
# 14
# FizzBuzz
FizzBuzz は判定の順番が命。15 の倍数は 3 の倍数でもあり 5 の倍数でもあるので、先に 15 から判定する。この順序を間違えると、たとえば 15 が "Fizz" になってしまう。
msub は「乗算した結果をもう一つの値から引く」を 1 命令でやる命令。剰余計算の常套句として覚えておくと便利。
ループカウンタを x19 (callee-saved) に置いたのは、printf 呼び出しで caller-saved レジスタ (x0-x18) が壊されても値が保持されるから。これをサボると、printf から戻ってきたときにカウンタの値が消えている、というバグになる。
